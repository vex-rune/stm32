# STM32 串口开发重大踩坑总结（终极经验教训）

## 一、本次问题现象

1. 代码刚烧录正常运行，重启后 ST-LINK 突然提示：`Target no device found` 找不到芯片。
2. 电脑设备、驱动、接线全部正常，但调试、烧录全部失败，芯片仿佛"锁死"。
3. 必须通过 BOOT 跳线 + 强制整片擦除才能救回芯片。

## 二、核心根本原因（最重要）

**错误写法：在 `while(1)` 死循环中使用了阻塞式串口接收**

`HAL_UARTEx_ReceiveToIdle()` 是阻塞式函数：

- 没有串口数据时，CPU 会永久停在这一行死等，卡死内核。

**结果：**

- CPU 不跑循环、不响应任何调试指令
- ST-LINK 无法 halt 内核 → 直接识别不到设备
- 程序霸占调试端口，看似"端口被占用"，实际是代码卡死 CPU

## 三、踩坑经验 & 技术教训

### 1. 主循环严禁使用任何阻塞式 HAL 函数

包括：串口阻塞接收、延时等待、I2C 阻塞读取、SPI 阻塞等待。

> 一旦卡死内核，ST-LINK 直接失联，芯片假死。

### 2. 串口接收必须使用「中断非阻塞」方式

| 方式 | 写法 | 说明 |
| --- | --- | --- |
| ❌ 错误 | `HAL_UARTEx_ReceiveToIdle()` | 阻塞卡死 |
| ✅ 正确 | `HAL_UARTEx_ReceiveToIdle_IT()` | 空闲中断接收 |

> 中断方式不占用 CPU，主循环永远运行，调试永远不会锁死。

### 3. 所有数据解析业务，绝对不放在中断里

- **中断**：只做标记、置位标志位
- **解析、打印、逻辑处理**：全部放 `while(1)`

### 4. `while(1)` 主循环必须留空、必须轻量

主循环一定要加 `HAL_Delay` 释放总线，否则 CPU 满载容易导致调试异常。

### 5. STM32 芯片并不是坏了，是被程序卡死

新手 90% 芯片连不上，都是：**代码阻塞内核导致调试接口失效**，不是硬件问题。

## 四、芯片救回紧急方案（卡死直接用）

1. **BOOT0 接 3.3V**，进入系统 ISP 模式
2. **CubeProgrammer**：100 kHz、软件复位连接
3. 执行 **Full chip erase** 整片擦除
4. **BOOT0 恢复 GND**，芯片复活

## 五、最终串口开发标准规范（永久记住）

1. 串口接收一律 **中断异步接收**，禁止阻塞等待
2. 主循环 **只判断标志位**，不卡、不等、不阻塞
3. 中断只做 **记录数据 + 开启下一次接收**
4. **业务逻辑全部放在 `while(1)` 中执行**

## 六、个人总结

> 本次最大收获：**STM32 最难排查的 Bug 往往不是硬件坏了，而是自己写的阻塞代码锁死了 CPU。**

以后写任何外设代码，优先保证：**主循环永远不卡死、不阻塞、不等待**。
